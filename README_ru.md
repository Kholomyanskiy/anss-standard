## Какую проблему решает этот стандарт

Я написал 12-строчное требование для проекта. Агент сгенерировал работающий код, который всё равно сломал производственную логику — потому что разрешил неявное допущение технически корректным, но архитектурно неправильным способом.

Классические стандарты ТЗ — IEEE 830, ISO/IEC 29148, ГОСТ 34.602 — создавались для человека-читателя, который терпит неоднозначность и заполняет пробелы из опыта. AI-агенты работают иначе. Они разрешают неопределённость *немедленно*, используя обучающие данные как ориентир.

**На проектах где я активно использовал ANSS, количество итераций сократилось с 5–7 до 2–3.**

ANSS — формат спецификации для **снижения ошибок, вызванных неоднозначностью** в AI-разработке. Не за счёт устранения суждения агента, а за счёт превращения неявных ограничений в явные.

## Как ANSS отличается от обычного ТЗ

```
Обычное ТЗ:                        ANSS:

Требования                         Требования
     ↓                                  ↓
AI-агент                          Слой [A] (правила агента)
     ↓                                  ↓
Заполняет пробелы из              Инварианты (явные ограничения)
обучающих данных                        ↓
     ↓                            Agent Review (аудит ТЗ)
Код                                     ↓
                                   Код
```

## До / После

**До ANSS:**

Требование: `"Держи приложение лёгким"`

Результат агента:
```
+ express@4.18.2
+ axios@1.4.0
+ lodash@4.17.21
+ uuid@9.0.0
```
Технически "лёгкое". Архитектурно неверно для проекта без зависимостей.

**После ANSS:**

```
INV-001: Без внешних npm-пакетов
Нельзя:  добавлять require() npm-модулей
Причина: приложение должно работать без npm install
Проверка: нет node_modules-импортов в server.js
```

Результат агента: только нативные модули Node.js.

## Почему существующих форматов недостаточно

| Возможность | Классический SRS/ТЗ | ANSS |
|-------------|---------------------|------|
| Читаемые требования для людей | ✓ | ✓ |
| Порядок чтения для агентов | ✗ | ✓ |
| Машиночитаемые ограничения | ✗ | ✓ |
| Проверяемые условия ограничений | ✗ | ✓ |
| Самопроверка перед написанием кода | ✗ | ✓ |
| Явные зоны "что НЕ менять" | ✗ | ✓ |
| Компас для неопределённых ситуаций | ✗ | ✓ |

## Для кого это

ANSS принесёт максимальную пользу если вы:

- Используете **Claude Code, Cursor, GitHub Copilot или Aider** для реальных проектов
- Наблюдали как агент делает что-то "разумное", что ломает то, что казалось очевидным
- Пишете ТЗ которые люди понимают правильно, а агенты интерпретируют по-своему
- Строите системы где неправильное неявное допущение имеет реальные последствия

## Ключевые механизмы

### Слой [A] — секции для агентов в первую очередь

```
[D] Domain        — ЧТО строить    → заказчик, PM, клиент
[E] Engineering   — КАК строить    → разработчик, архитектор
[A] Agent         — ПРАВИЛА АГЕНТА → AI-агенты читают это первым
```

Слой `[A]` содержит всё необходимое агенту: инварианты, принципы, глоссарий, операционные правила.

---

### Инварианты — явные ограничения с проверяемыми условиями

```
INV-001: Без внешних npm-пакетов
Нельзя:  добавлять require() npm-модулей
Причина: приложение должно работать без npm install
Проверка: нет node_modules-импортов в server.js
```

Поле "Проверка" — конкретное, верифицируемое условие. Не рекомендация.

---

### Архитектурные принципы — компас для неопределённых ситуаций

```
AP-001: Простое > Умное         AP-006: Fail Loud
AP-002: Монолит сначала         AP-007: Данные переживают код
AP-003: Купить до того как строить  AP-008: Скучные технологии побеждают
AP-004: Human Override всегда   AP-009: YAGNI
AP-005: Без vendor lock-in      AP-010: Обратимость важнее эффективности
```

Правила принятия решений, а не ценности. Применяются когда требования оставляют пробел.

---

### Agent Review — аудит спецификации до первой строки кода

```
✓ Найти противоречия между разделами
✓ Выявить пропущенные граничные случаи
✓ Проверить полноту критериев приёмки
✓ Убедиться в отсутствии конфликтов инвариантов
```

**Жёсткое правило: найдено более 3 проблем → остановиться и сообщить. Не продолжать.**

---

### Change Specification — безопасное изменение существующих систем

```
## Текущее состояние  — что существует сейчас
## Желаемое состояние — что должно стать
## Что НЕ менять     — явный список (критический раздел)
## Анализ влияния    — что ещё может быть затронуто
## План отката       — как отменить если что-то сломается
```

Раздел "Что НЕ менять" не существует ни в одном классическом стандарте. Именно здесь агенты чаще всего ломают проекты.

## Три уровня

```
CORE        15–20 стр.  80% проектов: боты, SaaS, API, автоматизации
EXTENDED    40–60 стр.  Безопасность, Compliance, детальное тестирование
ENTERPRISE  Полный стандарт  Банки, AI-платформы, регулируемые отрасли
```

Начинайте с Core. Большинству проектов этого достаточно.

## Быстрый старт — рабочая спецификация за 5 минут

1. Откройте `ANSS-Core-Template.ru.md`
2. Заполните **1.1** — что делает система и для кого (2 предложения)
3. Заполните **2.1** — глоссарий предметной области (5–10 терминов)
4. Напишите **2.5** — первые 3–5 инвариантов
5. Напишите **3.2** — пользовательские истории с критериями приёмки
6. Укажите **2.4** — технологический стек

Передайте спецификацию агенту:

```
Прочитай эту спецификацию. Запусти Agent Review (Раздел 14) до написания кода.
Сообщи обо всех противоречиях, пропущенных граничных случаях и конфликтах инвариантов.
Если найдёшь более 3 проблем — остановись и спроси. Не продолжай.
```

## Часто задаваемые вопросы

**Это реально предотвращает неправильные решения агентов?**
Ни один формат спецификации не устраняет суждение агента полностью. ANSS снижает ошибки вызванные неоднозначностью. Это не убирает необходимость человеческого ревью.

**Работает ли это с Claude Code / Cursor / Copilot?**
Да. Разделы с тегом `[A]` написаны так чтобы coding-агенты воспринимали их как абсолютную истину. Промпт Agent Review работает с любым агентом который следует структурированным инструкциям.

**Только для больших проектов?**
Core уровень покрывает 80% проектов — боты, скрипты, API-интеграции, одностраничные приложения. Если вы тратите больше 30 минут в неделю объясняя контекст агенту, ANSS окупается при первом использовании.

**Чем это отличается от обычного ТЗ или SRS?**
Обычное ТЗ описывает требования для человека. ANSS спроектирован для контекста где и человек-разработчик, и AI-агент читают один документ и должны прийти к одним выводам — включая самостоятельный аудит спецификации агентом до написания кода.
