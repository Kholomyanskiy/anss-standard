## Проблема

Классические ТЗ написаны для людей. Когда AI-агент их читает — он додумывает, галлюцинирует и нарушает инварианты.

ANSS даёт агентам ту же точность, что и разработчикам.

---

## Три слоя

Каждый раздел ANSS помечен слоем:

```
[D] Domain        — ЧТО строим   → Заказчик, PM, аналитик
[E] Engineering   — КАК строим   → Разработчик, архитектор
[A] Agent         — КАК это делает агент → AI-агенты (читают первым)
```

---

## Как работает

```
Человек пишет:                Агент читает:
──────────────                ─────────────
[D] Контекст (1.1)            ИНВАРИАНТЫ первыми  ← нельзя нарушать
[D] Глоссарий (2.1)           ПРИНЦИПЫ вторыми    ← компас в неопределённости
[A] Инварианты (2.5)    →     ГЛОССАРИЙ третьим   ← язык проекта
[A] Принципы (2.6)            Всё остальное
[D] User Stories (3.x)               ↓
[E] Архитектура (5.x)          Agent Review       ← ревью ТЗ до написания кода
                                     ↓
                               Реализация ✓
```

┌─────────────────────────────────────────────────────┐

**Начинай с Core.** Переходи выше только когда это реально нужно.

## Ключевые концепции

**Инварианты** — абсолютные ограничения, которые агент не может нарушить:
```
INV-001: Нет внешних npm пакетов
Нельзя: добавлять require() npm-модулей
Причина: приложение должно запускаться без npm install
Проверка: нет node_modules импортов в server.js
```

**Архитектурные принципы** — компас при неопределённости:
```
AP-001: Simple > Clever        AP-006: Fail Loud
AP-002: Monolith First         AP-007: Data Outlives Code
AP-003: Buy Before Build       AP-008: Boring Technology Wins
AP-004: Human Override Always  AP-009: YAGNI
AP-005: No Vendor Lock-In      AP-010: Reversibility Over Efficiency
```

**Agent Review** — обязательный шаг до написания кода:
```
Агент ищет: противоречия · пропущенные edge cases · недостающие AC · конфликты с инвариантами
Правило: если > 3 проблем → остановиться и спросить, не додумывать.
```

**Change Specification** — отдельный формат для изменений существующих систем:
```
Current State → Desired State → Что НЕ меняем → Impact Analysis → Rollback
```

## Быстрый старт

**5 минут — минимальное рабочее ТЗ:**

1. Заполни **1.1** — что строим и для кого (2 предложения)
2. Заполни **2.1** — глоссарий домена (5–10 терминов)
3. Напиши **2.5** — 3–5 инвариантов
4. Напиши **3.2** — user stories с Acceptance Criteria
5. Зафиксируй **2.4** — технологический стек

Затем передай ТЗ агенту и попроси выполнить **Agent Review (Раздел 14)** перед кодом.
