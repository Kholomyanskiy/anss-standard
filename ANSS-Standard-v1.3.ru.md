# AI-NATIVE SYSTEM SPECIFICATION STANDARD
## Персональный стандарт Артёма Холомянского — версия 1.3, 2026

---

## О ДОКУМЕНТЕ

Это не просто шаблон ТЗ. Это операционная система проекта — для людей и AI-агентов одновременно.

**Что внутри:** требования к системе, архитектурные решения, экономическое обоснование, правила работы агентов, инварианты которые нельзя нарушать.

**Основан на:** ГОСТ 34.602, IEEE 830, ISO/IEC 29148, ISO/IEC 25010, ISO/IEC 42001/42005, EU AI Act, GDPR, OWASP, C4 Model, Shape Up, ADR, SDD-методологии.

## МАРКИРОВКА СЛОЁВ

Каждый раздел помечен одним из трёх слоёв:

```
[D] Domain Specification    — ЧТО строим. Для заказчика, аналитика, PM.
                              Бизнес-цели, пользователи, требования, compliance.
```


*Примеры инвариантов:*

```
INV-001: Атомарность финансовых операций

INV-003: Неизменность аудит-лога
```

## 2.6 АРХИТЕКТУРНЫЕ ПРИНЦИПЫ [A] ✅

*Принципы — это не инварианты. Инвариант нельзя нарушить никогда.*
*Принцип — это руководство для принятия решений когда требований недостаточно.*
*Агент обращается к принципам при любой неопределённости.*

```
AP-001: Simple > Clever
Простое решение которое работает лучше умного которое сложно поддерживать.
При выборе между элегантным и понятным — выбирай понятное.

AP-002: Monolith First
Начинай с монолита. Микросервисы только когда монолит стал реальной проблемой.
Не проектируй для масштаба которого ещё нет.

AP-003: Buy Before Build
Сначала проверь существующие решения. Строй только то чего нет на рынке
или что критично для дифференциации продукта.

AP-004: Human Override Always Available
В любой AI-системе человек должен иметь возможность отменить решение агента.
Нет автономных необратимых действий без подтверждения человека.

AP-005: No Vendor Lock-In Without Justification
Зависимость от конкретного провайдера только с явным бизнес-обоснованием.
LLM, облако, сервисы — через адаптеры если замена реалистична.

AP-006: Fail Loud
Лучше явная ошибка чем тихая неправильная работа.
Агент сообщает о проблеме немедленно — не замалчивает и не обходит.

AP-007: Data Outlives Code
Код можно переписать. Данные потерять нельзя.
Любое решение затрагивающее данные требует особой осторожности.

AP-008: Boring Technology Wins
Проверенный стек лучше нового и блестящего.
Новую технологию вводить только если она решает конкретную проблему.

AP-009: YAGNI (You Aren't Gonna Need It)
Не добавляй функциональность "на будущее" без явной текущей потребности.
Агент не реализует то чего нет в спецификации.

AP-010: Reversibility Over Efficiency
При прочих равных — выбирай решение которое легче отменить.
Необратимые решения требуют дополнительного подтверждения.
```

*Как использовать:*
*Когда агент сталкивается с выбором не описанным в спецификации —*
*он проверяет принципы и принимает решение в соответствии с ними.*
*Если принципы конфликтуют — останавливается и спрашивает.*

*Принципы специфичны для проекта: удали неприменимые, добавь свои.*

```
Статус: Принято / Устарело

Решение: [что выбрали и почему]
Trade-offs: [с чем мирится]

```

*Decision Log — оперативные решения (почему выбрано конкретное решение):*
```
DEC-001: [название]
Дата: YYYY-MM-DD

Вопрос: [что решали]
Решение: [что выбрали]
Причина: [почему именно это]
Альтернативы отклонены: [что и почему нет]
```

*Примеры Decision Log:*
```
DEC-001: Выбор Redis вместо Memcached
Причина: нужны persistent keys и pub/sub для уведомлений.
Отклонено: Memcached — нет persistence, нет pub/sub.

DEC-002: Монолит вместо микросервисов
Причина: команда 2 человека, нет смысла в distributed overhead.
Отклонено: микросервисы — избыточны для текущего масштаба.
```

## 5.7 Architecture Fitness Functions 📋
*Применимо: Medium+ системы где важна проверяемость архитектурных характеристик*

*Fitness Functions — способ проверить архитектуру, а не просто описать её.*
*Не "система должна быть масштабируемой" а "выдерживает 1000 RPS без деградации".*

```
FF-001: Производительность под нагрузкой
Проверка: нагрузочный тест 1000 RPS в течение 10 минут
Критерий: p95 < 200ms, error rate < 0.1%
Когда: перед каждым релизом

FF-002: Изоляция LLM-зависимости
Проверка: замена провайдера LLM требует изменений максимум в 2 модулях
Критерий: grep -r "openai" | grep -v "adapter" | wc -l == 0
Когда: при добавлении нового LLM-вызова

FF-003: Скорость сборки
Проверка: время полного CI pipeline
Критерий: < 10 минут
Когда: при каждом коммите в main

FF-004: Покрытие тестами
Проверка: coverage report
Критерий: unit > 80%, integration > 60%
Когда: при каждом PR

FF-005: Добавление нового интегратора
Проверка: новый payment provider требует изменений только в adapter слое
Критерий: никаких изменений в business logic слое
Когда: при добавлении интеграции
```

## 15.3 Fail-Fast условия ✅

```
АГЕНТ ОСТАНАВЛИВАЕТСЯ И СПРАШИВАЕТ при:
□ Выборе между несколькими архитектурными подходами
□ Противоречии между требованиями
□ Задаче вне описанной спецификации
□ Изменении публичного API или схемы БД
□ Решение нарушает любой из ИНВАРИАНТОВ (Раздел 2.5)
□ Предложенное решение существенно увеличивает сложность
  без явной бизнес-необходимости (YAGNI violation)

АГЕНТ ДЕЙСТВУЕТ САМОСТОЯТЕЛЬНО при:
□ Выборе внутри реализации без внешних эффектов
□ Мелких деталях именования или форматирования
□ Решениях явно следующих из существующих паттернов
```

## 15.4 Мультиагентная разработка 📋

- PROJECT_STATUS.md — Orchestrator ведёт, обновляет после каждой подзадачи

## 15.5 Живая спецификация ✅

- Спецификация обновляется в той же итерации что и код
- Кто обновляет: человек (агент предлагает → человек принимает)
- Изменение в коде без обновления спецификации = технический долг
- Версионирование в Git вместе с кодом

## 15.6 Agent Review Specification ✅

*Обязательный шаг ДО начала написания кода.*
*Агент делает ревью ТЗ — не кода.*

```
Агент проверяет:
□ Противоречия между разделами ТЗ
□ Пробелы (неописанные ситуации которые точно возникнут)
□ Edge cases которых нет в спецификации
□ Acceptance Criteria которых не хватает
□ Конфликты с ИНВАРИАНТАМИ (Раздел 2.5)
□ Требования которые технически нереалистичны

Если найдено > 3 проблем:
→ Остановиться и сообщить ПЕРЕД написанием кода
→ Не угадывать, не заполнять пробелы самостоятельно

Промпт для Agent Review:
"Прочитай ТЗ и найди:
1. Противоречия между требованиями
2. Ситуации не описанные но которые возникнут
3. Acceptance Criteria которых не хватает
4. Edge cases отсутствующие в спецификации
5. Конфликты с инвариантами из Раздела 2.5
НЕ предлагай решения — только находи проблемы."
```

## 15.7 Чеклист качества ТЗ ✅

*Агент проходит перед сдачей. > 3 не выполнены → доработать.*

```
КОНТЕКСТ И ЦЕЛЬ
□ Цель написана так что понятно ЗАЧЕМ, а не только ЧТО
□ Все термины домена есть в GLOSSARY.md
□ ТЗ не противоречит ARCHITECTURE.md
□ Инварианты системы явно перечислены (Раздел 2.5)

ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ
□ Описан полный user flow — happy path + все отклонения
□ Все бизнес-правила явные, нет допущений "очевидно что..."
□ Описано поведение при каждой ожидаемой ошибке
□ Edge cases перечислены с ожидаемым поведением

ТЕХНИЧЕСКИЕ ТРЕБОВАНИЯ
□ API Contracts: все эндпоинты + все коды ошибок + форматы
□ Data Models описаны — агент не принимает решения сам
□ NFR конкретные и измеримые

UI (если применимо)
□ Описаны все 4 состояния: Loading / Empty / Error / Success
□ Валидация: правила + когда показывать ошибки

ACCEPTANCE CRITERIA
□ Каждый критерий однозначно проверяемый
□ Все требования покрыты хотя бы одним критерием
□ Есть критерии проверки инвариантов

ПОЛНОТА
□ Что НЕ входит в scope — написано явно
□ Зависимости между модулями явные
□ Compliance определены (или явно N/A)
□ Forbidden Patterns написаны

КАЧЕСТВО
□ Нет противоречий между разделами
□ Agent Review выполнен (Раздел 15.6)
□ Decision Log заполнен для нетривиальных решений
```
