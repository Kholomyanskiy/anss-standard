# РАЗДЕЛ 1. КОНТЕКСТ И ЭКОНОМИКА

## 1.1 Суть проекта

**Что строим:**
Операционная система для сети из 3 ломбардов — заменяет Excel в ежедневной работе сотрудников и даёт владельцу зводную картину сети в реальном времени через Telegram.

**Проблема которую решаем:**
Сеть ведёт операционный учёт в Excel вручную: владелец не видит сводку по сети, нет контроля движения инкассированных изделий, есть реальный риск злоупотреблений со стороны персонала (хищение кассы, манипуляции с весами, «призрачный» выкуп).

**Почему сейчас:**
Бизнес растёт (3 отделения), ручной контроль не масштабируется. Каждый инкассированный предмет — замороженные деньги без трека; риск потери выше с каждым месяцем.

**Что НЕ делаем в этой версии:**
- Не заменяем PawnExpert — он остаётся основной программой для оформления залогов (НБУ / Держфінпослуг)
- Не дублируем ведение договоров с клиентами
- Не подключаемся к PawnExpert через API (его нет) — читаем экспортные .xls файлы
- Не переносим архивные данные из Excel — система стартует с текущего момента
- Не делаем мобильное приложение (PWA/нативное) — только мобильный веб

## 1.2 Цели и критерии успеха

**Бизнес-цели:**
- Владелец знает статус каждого инкассированного изделия в реальном времени (не раз в неделю по Excel)
- Ежедневная сверка кассы делается нажатием одной кнопки, не вручную
- Каждое злоупотребление — кража кассы, занижение веса при скупке, «призрачный» выкуп — оставляет след в системе

**KPI — как поймём что успех:**
- Владелец получает утреннее сводное сообщение каждый день в 9:00 (целевое: 100% дней)
- Время сверки кассы: < 3 минут в день (сейчас ~30 минут)
- Ни одно инкассированное изделие не «теряется» — 100% трек по статусам
- Сотрудники переходят на систему и прекращают вести Excel в течение 2 недель после запуска

**Definition of Done проекта:**
1. Все 4 модуля задеплоены и работают в production
2. Все 3 отделения загрузили реальные .xls файлы и получают утреннее сводное
3. Сотрудники прошли онлайн-обучение (до 1 часа)
4. Первые 2 недели работы без критических багов

# РАЗДЕЛ 2. ПРЕДМЕТНАЯ ОБЛАСТЬ

## 2.1 Глоссарий

**Залог**
Определение: предмет (украшение, техника), переданный клиентом ломбарду под выдачу кредита. Оформляется в PawnExpert.
НЕ путать с: «скупкой» — где клиент продаёт предмет навсегда.
В коде: таблица `pledges`, статус `active`.

**Инкассация**
Определение: переход залога на баланс ломбарда, если клиент не пришёл до истечения срока договора.
НЕ путать с: «выкупом» — когда клиент сам забирает предмет, заплатив долг.
В коде: смена статуса `pledges.status`: `active` → `on_branch`. Событие, от которого начинается трек движения изделия.

**Портфель залогов**
Определение: расчётная сумма всех действующих залогов по отделению = `SUM(loan_amount) WHERE status='active' AND branch_id=X`. Не хранится как поле — вычисляется динамически.
НЕ путать с: реальной кассой. Портфель — деньги «в работе» у клиентов.
В коде: вычисляемый агрегат, не колонка.

**Кошелёк П/О (пункт отделения)**
Определение: физическая наличность в кассе отделения.
НЕ путать с: «Домашним» — деньги на физлице владельца.
В коде: `cash_wallet` с типом `branch`.

**Домашний (кошелёк)**
Определение: деньги на физическом лице владельца. Используется для выкупа просроченных залогов от имени физлица, чтобы не держать золото на балансе юрлица (налоговая оптимизация).
В коде: `cash_wallet` с типом `personal`.

**Скупка**
Определение: операция где клиент продаёт предмет ломбарду навсегда (не залог).
НЕ путать с: «инкассацией» (залог не выкуплен).
В коде: таблица `purchases`.

**Замороженные средства**
Определение: сумма к выкупу всех инкассированных предметов = `SUM(redemption_amount) WHERE status IN ('on_branch','hold')`. Деньги вложены, выручка ещё не получена.
В коде: вычисляемый агрегат.

**Скупщик**
Определение: внешний покупатель инкассированных изделий (Покупатель А — золото, Покупатель Б — серебро). После отправки ожидается подтверждение оплаты.
В коде: таблица `buyers`.

**Залоговая ставка**
Определение: % в день, который начисляется на сумму залога с момента выдачи. Из PawnExpert.
В коде: `pledges.interest_rate`.

## 2.2 Пользователи

**Роль: Владелец**
- Кто это: собственник сети, 1 человек
- Что делает в системе: получает утреннее сводное, подтверждает маршрут каждого инкассированного изделия, подтверждает оплаты от скупщиков, смотрит аналитику в веб-дашборде
- Что ценит: скорость принятия решений, контроль
- Ожидаемое количество: 1
- Частота использования: daily (Telegram), weekly (дашборд)
- Каналы: Telegram-бот + веб-дашборд (desktop)

**Роль: Сотрудник (касир)**
- Кто это: сотрудник отделения, 2 человека на отделение × 3 отделения = 6 человек
- Что делает в системе: вносит кассовые операции, фиксирует инкассации, скупку, продажи, открывает/закрывает смену
- Что ценит: простота, быстрый ввод с мобильного
- Ожидаемое количество: 6
- Частота использования: daily
- Видит: только своё отделение

**Роль: Администратор**
- Кто это: владелец или техническое лицо (совпадает с владельцем на старте)
- Что делает в системе: добавляет/деактивирует сотрудников, настраивает справочники (отделения, скупщики, диапазоны цен, ставки бонусов), маппинг колонок PawnExpert
- Ожидаемое количество: 1
- Частота использования: monthly

## 2.3 Ограничения и допущения

**Технические ограничения (нельзя менять):**
- PawnExpert не имеет API — интеграция только через .xls экспорт

- Бесплатные tier Supabase/Railway — без гарантированного SLA провайдера

**Бизнес-ограничения:**
- Бюджет: удалён из публичной версии
- Дедлайн: нет жёсткого
- Команда: Артём Холомянский + Claude Code

**Допущения:**
- У клиента стабильный интернет в каждом отделении
- Сотрудники используют смартфон с Chrome/Safari ежедневно
- Клиент предоставит реальный .xls файл из PawnExpert до старта Этапа 1 (критичное условие)

## 2.5 ИНВАРИАНТЫ ✅

```
INV-001: Фото обязательно при инкассации
Нельзя: сохранить запись инкассации без прикреплённого фото изделия.
Причина: фото — единственное доказательство что изделие существует и принято в нужном состоянии.
Проверка: форма/бот не отправляются без file_id фото. Backend отклоняет запрос без photo_url.

INV-002: Уникальность залога
Нельзя: создавать два записи pledges с одинаковыми contract_num + branch_id.
Причина: дублирование = двойной учёт, искажение Портфеля.
Проверка: UNIQUE constraint на (contract_num, branch_id) в БД.

INV-003: Операции нельзя удалять
Нельзя: удалять любые внесённые операции (касса, инкассация, скупка, продажа, смена).
Причина: аудит-лог должен быть полным и неизменяемым.
Проверка: нет DELETE endpoint'ов для операций. Только флаг is_active=false (soft delete) с записью кто деактивировал и почему.

INV-004: Маршрут изделия подтверждает только владелец
Нельзя: менять статус инкассированного изделия на направление (→ скупщик / → физлицо / → [Город] / → магазин) без нажатия кнопки владельцем в Telegram.
Причина: контроль от злоупотреблений — сотрудник не решает куда уходит изделие.
Проверка: смена статуса на routing-статус требует owner_approved=true в таблице pledges.

INV-005: Цена скупки в диапазоне
Нельзя: провести скупку металла по цене за грамм вне настроенного диапазона без подтверждения владельца.
Причина: защита от занижения цены сотрудником (разница в карман).
Проверка: backend проверяет price_per_gram ∈ [min_price, max_price] по справочнику. Если нет — запрос блокируется до получения owner_approval_token.
```

## 2.6 АРХИТЕКТУРНЫЕ ПРИНЦИПЫ ✅

```
AP-001: Simple > Clever
  — Vanilla JS вместо сложного React если сохраняет простоту развертывания

AP-002: Monolith First
  — Один FastAPI сервис, не разбивать на микросервисы

AP-003: Buy Before Build
  — Supabase, Railway, Netlify — не поднимать свой PostgreSQL/S3

AP-004: Human Override Always Available
  — Владелец всегда может вручную изменить статус через админ-панель

AP-005: No Vendor Lock-In Without Justification
  — Данные в PostgreSQL (стандартный SQL), фото в Supabase Storage с возможностью миграции

AP-006: Fail Loud
  — Ошибки парсинга .xls — явное сообщение с деталями, не молчаливый пропуск

AP-007: Data Outlives Code
  — Soft delete везде. Ничего не теряется навсегда.

AP-008: YAGNI
  — Не строить для гипотетических 100 отделений — система для 3-х.

AP-009: Mobile First
  — Веб-приложение сотрудника проектируется для телефона (Chrome/Safari).
  — Desktop — вторично для сотрудника, первично для дашборда владельца.

AP-010: One Source of Truth per Entity
  — Портфель залогов вычисляется из pledges, не хранится отдельно.
  — Статус изделия — одно поле pledges.status, не дублируется.
```

## 3.3 Детальные требования по модулям

```
МОДУЛЬ: Telegram-бот (Модуль 1)
Назначение: уведомления владельцу и сотрудникам, подтверждения маршрутов и оплат
Связанные инварианты: INV-001, INV-004

User Flow (утреннее сводное):
1. Cron в 9:00 → агрегирует данные по всем отделениям из PostgreSQL
2. Формирует одно сообщение со сводкой + inline-кнопки отделений
3. Отправляет владельцу в Telegram

User Flow (инкассация):
1. Сотрудник пишет /incasso 526105
2. Бот ищет contract_num=526105 в pledges для branch_id сотрудника
3. Если найдено — показывает данные, запрашивает фото
4. Сотрудник отправляет фото → бот загружает в Supabase Storage
5. Запись обновляется: статус active→on_branch, photo_url заполнен
6. Владелец получает уведомление с фото + кнопки маршрута

Бизнес-правила:
BR-001: Один бот — две роли
  Telegram ID → определяет роль (owner / staff + branch_id)
```

## 5.1 Обзор

```
    └──▶ Web Dashboard (владелец: аналитика, отчёты)
```

**Слои приложения:**
```
API Layer (FastAPI routers)    — HTTP endpoints, auth, validation

Repository Layer               — запросы к БД (asyncpg/SQLAlchemy)
```

- Telegram Layer использует Service Layer — не обращается к БД напрямую
- Бизнес-логика (расчёты, проверка инвариантов) — только в Service Layer
- Прямые запросы к БД — только через Repository Layer

## 5.4 ADR + Decision Log

```
ADR-001: Один бот вместо двух
Контекст: нужны уведомления и для владельца и для сотрудника.
Решение: один бот, роль по Telegram ID.
Причина: проще управлять, один токен, один деплой.
Trade-offs: немного сложнее логика определения роли.
Для агентов: всегда проверяй telegram_id перед показом меню.

ADR-002: Supabase Storage вместо Google Drive
Контекст: нужно хранить фото инкассаций.
Решение: Supabase Storage, приватный bucket.
Причина: нет зависимости от Google-аккаунта, единая платформа с PostgreSQL,
  signed URL генерируется бэкендом (TTL 1 час).
Trade-offs: миграция фото потребует скрипта если уйдём с Supabase.

ADR-003: Портфель как вычисляемый агрегат
Контекст: нужно показывать «Портфель залогов» в реальном времени.
Решение: вычислять SUM(loan_amount) WHERE status='active' при каждом запросе.
Причина: исключает рассинхронизацию при ошибках обновления денормализованного поля.
Trade-offs: дополнительная нагрузка на БД, допустимо при ~1500 записей/год.
```

```
DEC-001: Маппинг колонок PawnExpert в БД (не в коде)
Вопрос: как обновлять маппинг когда PawnExpert обновляется?
Решение: таблица column_mappings, редактируется через админ-панель.
Причина: не нужен деплой при изменении структуры .xls.
Отклонено: хардкод в коде — требует деплой при любом изменении.
```

# РАЗДЕЛ 9. УПРАВЛЕНИЕ ПРОЕКТОМ

## 9.1 Декомпозиция

| Модуль | Зависит от | Кто | Оценка |
|---|---|---|---|
| БД схема + миграции | — | Артём + Claude | 2 дня |
| FastAPI: auth, base | БД | Артём + Claude | 2 дня |
| Парсер .xls + upload API | БД | Артём + Claude | 3 дня |
| Telegram-бот MVP (сводное + алерты) | Парсер | Артём + Claude | 5 дней |
| Трек изделия (инкасс + маршрут) | Бот MVP | Артём + Claude | 4 дня |
| Подтверждение скупщика | Трек | Артём + Claude | 2 дня |
| Веб-приложение: все вкладки | API | Артём + Claude | 10 дней |
| Расчёт зарплаты + табель | Веб | Артём + Claude | 3 дня |
| Отчёт по скупщикам | Веб | Артём + Claude | 2 дня |
| Веб-дашборд владельца | Все | Артём + Claude | 7 дней |

**Критический путь:** БД → API auth → Парсер .xls → Telegram-бот → Трек изделия → Веб-приложение

## 9.2 Риски

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| Клиент не предоставляет .xls вовремя | M | H | Старт невозможен без .xls — закреплено в договоре |
| PawnExpert меняет формат .xls при обновлении | M | M | Маппинг в БД — не требует деплоя (DEC-001) |
| Railway/Supabase free tier вводят ограничения | L | H | Миграция на платный план (~$20/мес) |
| Сотрудники не переходят на систему | M | H | Онлайн-обучение + 2-недельная поддержка |
| Telegram API блокировка | L | H | Fallback — веб-дашборд (уведомления придут позже) |

## 9.3 Что явно вне scope

- Замена PawnExpert или дублирование его функций
- Мобильное приложение (iOS/Android native)
- Интеграция с банковскими API
- Автоматическое получение курсов металлов (вводится вручную)
- Перенос архивных данных из Excel
- Расширение на 4+ отделения (требует Change Request)
- Система напоминаний клиентам о сроке залога (не в scope MVP)
